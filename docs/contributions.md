# contributions

## Why Contributions Matter

`elastic space` gets stronger when other people contaminate it in interesting ways.

But open submission only works if there is a clear social and technical structure. Otherwise the place turns into a junk drawer with a loading spinner.

## Contribution Types

Contributors should be able to submit:

- fully built worlds
- partial worlds that need finishing
- visual assets
- sound pieces
- writing
- code toys
- secret route ideas
- seasonal event packs

## Submission Model

Each submission should include:

- title
- creator name
- short intent statement
- content warning notes if needed
- asset list
- technical requirements
- preferred mood tags
- whether the piece wants stable links, drifting links, or hidden-only access

This helps curation without forcing artists to write grant proposals about their haunted GIF page.

## Curation Rules

The curation layer should ask:

- Is it alive?
- Is it authored?
- Is it distinct from what already exists?
- Does it add a new corridor to the maze?
- Does it function well enough to be experienced as intended?

The goal is not polish in the corporate sense. The goal is conviction.

## Technical Contract

Every contribution should eventually conform to a simple contract:

- a world definition file
- a content entry point
- a declared asset folder
- a metadata block
- optional custom navigation hints
- admin-visible ownership and editing metadata

That gives the site enough consistency to ingest diverse work without sanding the edges off it.

## Roles

The project can support a few lightweight roles:

- `core curator`: decides what enters active rotation
- `world builder`: creates full experiences
- `signal booster`: assembles playlists, spotlights, and temporary routes
- `guest invader`: contributes a piece for a limited run

## Rotation Strategy

Not every contribution should appear equally.

Some should:

- stay permanent
- appear only during certain windows
- emerge after specific page sequences
- resurface occasionally as relics
- vanish after a run, leaving traces behind

Scarcity makes discovery matter.

## Suggested Next System

After the manifesto stage, the next concrete artifacts should be:

1. a `world` file format
2. a starter template for making a new world
3. an intake form for contributors
4. a first drift algorithm
5. a curation dashboard, even if crude

That is the point where the project becomes operational instead of merely seductive.
